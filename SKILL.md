# Karpathy LLM Wiki

Build and maintain a shared LLM wiki through GitHub pull requests. The skill prepares small, reviewable wiki changes; it does not merge them. It supports both CI-driven PR creation and local PR creation for teams that cannot put API keys in CI.

Core ideas:
- The wiki is a persistent, compounding artifact.
- Source material stays immutable; wiki articles improve over time.
- Every write is scoped to the configured wiki root and is prepared for human review.

## Architecture

Choose a **wiki root** before any write operation:
- Default: `wiki/`
- Alternate common choice: `llm/wiki/`
- If the user names another root, use that root for the current wiki.

Persist the choice in `<wiki-root>/llm_wiki_config.md`. On later tasks, discover the wiki root from this marker file and the required sibling structure. Do not rely on conversation memory alone.

Important: before the first initialization, `llm/wiki/` is not discoverable unless the user explicitly names it. If no wiki exists and the user does not name a root, initialize the default `wiki/`.

All write operations must stay under the configured wiki root. Do not modify repo code, app source, CI config, package manifests, or other files outside the wiki root when performing wiki work.

Inside the wiki root:

**raw/** — Immutable source material. Read existing files, but never rewrite or delete source files during ingest. Organized by topic subdirectories.

**articles/** — Durable wiki articles compiled from raw sources. Organized by topic subdirectories, one level only: `articles/<topic>/<article>.md`.

**index.md** — Global index. One row per article, grouped by topic, with link, summary, and Updated date.

**log.md** — Single canonical append-only operation log. Users do not edit this file directly; the skill appends structured entries as part of PR-safe changes.

**references/** — Template copies for the wiki project. Use bundled templates from this skill as the source when initializing or refreshing these files.

The skill's bundled `references/` directory contains the exact formats for raw files, articles, memory-bank archive pages, index entries, log entries, and compact summaries.

### Wiki Root Resolution

Resolve the wiki root in this order:

1. **Explicit user path** — If the current request names a root such as `wiki/`, `llm/wiki/`, or another path, use it for the current operation.
2. **Existing marker** — Search the repository for `llm_wiki_config.md` files with `Schema: collaborative-llm-wiki/v1`, prioritizing likely paths such as `wiki/llm_wiki_config.md` and `llm/wiki/llm_wiki_config.md`. A valid marker sits next to `raw/`, `articles/`, `index.md`, and `log.md`.
3. **Single valid structure** — If exactly one directory contains `raw/`, `articles/`, `index.md`, and `log.md`, use it as the root even if the marker is missing, then add `llm_wiki_config.md` on the next write operation.
4. **First setup or first ingest** — If no wiki exists and the user did not name a root, use the default `wiki/`. Do not infer `llm/wiki/` from the alternate-root example alone.
5. **Ambiguous roots** — If multiple valid roots exist and the user did not choose one, stop and ask which root to use. Do not write.

For CI usage, pass the root in the task prompt or environment-specific wrapper, for example: "Use wiki root `llm/wiki/`." The skill itself still verifies the root before writing.

### Initialization

Initialize only during an explicit setup or first ingest. Create missing files and directories under the configured wiki root:

- `<wiki-root>/llm_wiki_config.md` using `references/llm-wiki-config-template.md`
- `<wiki-root>/raw/` with `.gitkeep`
- `<wiki-root>/articles/` with `.gitkeep`
- `<wiki-root>/index.md` using `references/index-template.md`
- `<wiki-root>/log.md` with heading `# Wiki Log`
- `<wiki-root>/references/` with copies of the bundled templates when the environment supports copying files

If Query, Lint, PR-lint, or Compact cannot find the wiki structure, tell the user to initialize the wiki root first. Do not create the structure during read-only operations.

## Collaboration Principle

Prepare PR-safe changes only.

- Do not merge PRs.
- Assume one maintainer or owner performs the final merge.
- Keep each PR small enough for human review.
- Keep writes inside the configured wiki root.
- Treat `<wiki-root>/log.md` as an append-only operational file managed by the skill.
- Put broad or uncertain changes in `Follow-up Required` instead of editing many files.

## Execution Workflows

### CI PR Workflow

Use this when the team can provide the required API key to CI.

1. CI receives a wiki task, such as ingest, archive, lint, or compact.
2. CI runs the agent against the configured wiki root.
3. CI creates a branch and PR containing wiki-root-only changes.
4. PR-lint verifies the wiki boundary, log entry, and changed files.
5. A maintainer reviews and merges.

### Local PR Workflow

Use this when the team cannot put API keys in CI.

1. The user runs the agent locally.
2. The agent creates wiki-root-only changes.
3. The user opens a PR with those changes.
4. PR-lint verifies the wiki boundary, log entry, and changed files.
5. A maintainer reviews and merges.

Both workflows produce the same wiki files and log format.

## Preflight For Write Operations

Before Ingest, Archive, Lint with auto-fix, Compact, or approved log deletion:

1. Identify and validate the configured wiki root using Wiki Root Resolution.
2. Run `git status --short` and inspect changed paths.
3. Confirm all planned writes are under the wiki root.
4. Search planned target files for conflict markers: `<<<<<<<`, `=======`, `>>>>>>>`.
5. If conflict markers or unexpected existing edits affect a planned target, stop and report. Do not auto-resolve.

## Ingest

Fetch one source into `<wiki-root>/raw/`, then compile it into a wiki article under `<wiki-root>/articles/`. Keep the default PR scope to one source, one primary article, touched index entry, and one log entry.

### Fetch Raw Source

1. Get the source content using available web, file, or pasted-text tools.
2. Search `<wiki-root>/raw/` for the same URL, title, or obvious source identifier.
   - If a duplicate exists, do not create a second raw file. Report the duplicate and use the existing raw file if the user still wants an article update.
3. Pick a topic directory. Reuse an existing topic when it fits; create a new topic only for a distinct subject.
4. Save as `<wiki-root>/raw/<topic>/YYYY-MM-DD-descriptive-slug.md`.
   - Slug from source title, kebab-case, max 60 characters.
   - Published date unknown -> omit the date prefix from the file name. The metadata Published field still appears; set it to `Unknown`.
   - If a file with the same name already exists and it is not the same source, append a numeric suffix.
   - Preserve original text. Clean formatting noise. Do not rewrite opinions.

See `references/raw-template.md`.

### Compile Into A Wiki Article

Determine where the source belongs:

- **Same core thesis as an existing article** -> update that article with a focused patch.
- **New concept** -> create a new article under the most relevant topic.
- **Spans multiple topics** -> update the primary topic only, and record cross-topic effects in `Follow-up Required`.

Check for factual conflicts. If the new source contradicts existing article content, annotate the disagreement with source attribution inside the touched article.

See `references/article-template.md`. Key points:
- Sources field: author, organization, or publication name plus date, semicolon-separated.
- Raw field: markdown links to raw files.
- Relative paths from `articles/<topic>/` to raw files use `../../raw/<topic>/<file>.md`.

### Cascade Policy

Limit automatic cascade updates.

- Same-topic, directly affected article: may update if the patch is small and reviewable.
- Cross-topic article: do not edit automatically. Add a `Follow-up Required` item to the log entry.
- Broad rewrite, uncertain relation, or many touched files: report as follow-up.

### Post-Ingest

Update `<wiki-root>/index.md` only for touched articles. Do not rewrite the whole index.

Append one PR-level entry to `<wiki-root>/log.md`. Include the raw source, touched article, index update, decisions, and follow-up items. Use `references/log-entry-template.md`.

When preparing a commit message, include:

```text
Wiki-Log: <wiki-root>/log.md
Wiki-Change-ID: <change-id>
```

## Query

Search the wiki and answer questions without writing files.

Examples:
- "What do we know about X?"
- "Summarize everything related to Y in the wiki"
- "Compare A and B based on the shared wiki"

Steps:

1. Read `<wiki-root>/index.md` to locate relevant articles.
2. Read those articles and synthesize an answer.
3. Prefer wiki content over model memory.
4. Cite wiki articles with project-root-relative markdown links, such as `[Article Title](wiki/articles/topic/article.md)` or `[Article Title](llm/wiki/articles/topic/article.md)`.
5. Do not write files unless the user asks to archive or save the final answer.

### Archive / Memory Bank

Use Archive as the wiki's memory bank for conversation-derived knowledge: final answers, user-corrected conclusions, decisions, operating rules, reusable prompts, and "next time answer this way" notes.

When the user explicitly asks to archive, save, remember, or add an answer to the memory bank:

1. Archive only the revised final answer. If the user corrected the answer, include the correction and do not preserve the earlier draft as authoritative.
2. Create a new memory-bank archive page under `<wiki-root>/articles/<topic>/`.
3. Use `references/archive-template.md`.
4. Search for similar archive pages first to avoid duplicates.
5. Update only the touched entry in `<wiki-root>/index.md`; prefix the summary with `[Memory]`.
6. Append the archive action to the PR-level `<wiki-root>/log.md` entry, or create a new PR-level entry if archive is the only operation.

Archive pages are point-in-time memory snapshots. They do not come from raw source ingestion, do not create raw files, and do not trigger cascade updates. If the archived answer reveals that an existing article should change, record that in `Follow-up Required`.

## Lint

Use two lint modes: general lint and PR-lint.

### General Lint

General lint checks wiki health. It may auto-fix small, deterministic issues when the change stays reviewable and inside the wiki root.

Required deterministic checks:

- **Broken links** — markdown links in articles must point to existing files.
- **Missing index entries** — article files must appear in `<wiki-root>/index.md`.
- **Duplicate raw source** — raw files must not represent the same URL or obvious source identifier.
- **Conflict markers** — no wiki file may contain `<<<<<<<`, `=======`, or `>>>>>>>`.
- **Stale `[MISSING]` entries** — index entries marked missing must still be missing.

Auto-fix only when the fix is obvious and small. If many files would change, if a target is ambiguous, or if judgment is required, report the issue instead of editing.

Heuristic checks are always report-only:

- Factual contradictions across articles
- Outdated claims superseded by newer sources
- Missing conflict annotations where sources disagree
- Orphan pages with no inbound links
- Missing cross-topic references
- Concepts frequently mentioned but lacking a dedicated page
- Memory-bank archive pages whose cited articles have materially changed

When raw/article mismatches, article contradictions, or materially changed cited articles are found, also check memory-bank archive pages that cite the affected articles. Archive pages are point-in-time memory snapshots, so do not auto-edit them. Report mismatches as `Follow-up Required` with one recommended retention path: `keep`, `supersede`, or `delete-candidate`.

Record lint results and follow-ups in the PR-level log entry.

### PR-Lint

PR-lint is a merge-readiness check. It is report-only and does not auto-fix.

Check:

1. Every changed file is under the configured wiki root.
2. `<wiki-root>/log.md` has no conflict markers.
3. The new log entry is append-only.
4. The new `Change-ID` exists and is not duplicated.
5. Raw sources, touched articles, and index updates named in the log entry match the actual diff.
6. The PR has or is ready to use the commit trailers:
   - `Wiki-Log: <wiki-root>/log.md`
   - `Wiki-Change-ID: <change-id>`

If any check fails, report it as a blocking issue for the user or maintainer.

## Log Format

`<wiki-root>/log.md` is the single canonical append-only log.

Rules:

- Append one structured entry per PR by default.
- A PR with ingest, lint, and archive work may still use one entry if it is one coherent wiki change.
- Every entry must include a stable `Change-ID`, such as `chg-20260514-001`.
- `Commit` defaults to `linked by commit trailer`; do not write the current commit hash into the same commit's log entry.
- Users should not manually edit the log. If the log contains conflict markers or suspicious edits, PR-lint reports them and stops.

Use `references/log-entry-template.md`.

---

## Log Compact

Compact summarizes old detailed entries in `<wiki-root>/log.md`. Run compact only when the user asks.

Default compact scope is monthly. If the user asks for weekly, quarterly, topic-based, or PR-based compact, use that scope.

Compact behavior:

1. Read the selected detailed entries.
2. Resolve related commit IDs from git history by `Wiki-Change-ID` trailers and log history.
3. Add or update a compact summary in `<wiki-root>/log.md`.
4. Include each entry's `Change-ID`, operation, short summary, related commit IDs, and retention status.
5. Do not delete detailed entries during compact.

Retention statuses:

- `keep` — default.
- `delete-candidate` — may be removed if the user approves.
- `delete-approved` — user approved removal; deletion can be prepared in a separate PR-safe change.
- `deleted` — detailed entry was removed, but the compact summary and git history preserve the audit trail.

Only remove detailed entries when the user explicitly approves specific `Change-ID` values or exact log sections.

Use `references/log-compact-template.md`.

## Conventions

- Standard markdown with relative links throughout.
- Article topics are one level deep: `articles/<topic>/<article>.md`.
- Raw topics are one level deep: `raw/<topic>/<source>.md`.
- Today's date is used for Collected, Updated, Archived, and log Date fields unless source metadata gives a different Published date.
- Inside article files, links are relative to the current article.
- In conversation output, links are project-root-relative.
- All write operations must stay under the configured wiki root.
